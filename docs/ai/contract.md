## 💬 ПРАВИЛА КОММЕНТАРИЕВ

> **ЗАПРЕТ НА ЛЮБЫЕ КОММЕНТАРИИ КРОМЕ ТЕХНИЧЕСКИХ PHPDOC (БЕЗ ОПИСАНИЯ ЛОГИКИ) И ПОДАВЛЕНИЙ СТАТИЧЕСКОГО АНАЛИЗА.**

**TL;DR**: в коде разрешены только технические PHPDoc (без описания логики) и подавления статического анализа. Все остальные случаи СТРОГО ЗАПРЕЩЕНЫ.

Если ты всё же считаешь, что обычный комментарий (не PHPDoc и не подавление статического анализа) необходим:

1.  Не пиши его в коде.
2.  Добавь запись в файл `COMMENTS.MD` с указанием:
    - Файл и номер строки.
    - Предлагаемый текст комментария.
    - Обоснование: ЗАЧЕМ он нужен.
    - Проверка на очевидность ("капитанский" комментарий).

**РАЗРЕШЕНО (исключения):**
Эти типы комментариев фиксировать в COMMENTS.md НЕ НУЖНО.

1.  **PHPDoc**:
    - **РАЗРЕШЕНО**: технические аннотации типов/статического анализа, НО СТРОГО без описания логики.
    - **ЗАПРЕЩЕНО**: дублировать нативные типы и описывать параметры (`@param int $id ID пользователя` - ЗАПРЕТ).
2.  **Подавление статического анализа**:
    - **РАЗРЕШЕНО**: технические комментарии для подавления ошибок инструментов статического анализа, если исправление невозможно:
      - **PHPStan**: `@phpstan-ignore-line`, `@phpstan-ignore-next-line`, `@phpstan-ignore method.notFound` - только в PHPDoc.
      - **PHPCS**: `phpcs:ignore [Sniff.Name]`, `phpcs:disable [Sniff.Name]` ... `phpcs:enable [Sniff.Name]`, `phpcs:ignoreFile`.
      - **Rector**: `@noRector` - только в PHPDoc.
      - Примеры:
        - PHPStan: `@phpstan-ignore-line`, `@phpstan-ignore-next-line`, `@phpstan-ignore argument.type`
        - PHPCS: `phpcs:ignore Squiz.Arrays.ArrayDeclaration.SingleLineNotAllowed`, `phpcs:disable Generic.Commenting.Todo.Found` ... `phpcs:enable Generic.Commenting.Todo.Found`, `phpcs:ignoreFile`
        - Rector: `@noRector \Rector\DeadCode\Rector\ClassMethod\RemoveUnusedParameterRector`
    - Перед подавлением сначала попробуй исправить код, если это возможно в рамках архитектуры.
3.  **Code Coverage**:
    - **РАЗРЕШЕНО**: `@codeCoverageIgnore`, `@codeCoverageIgnoreStart` и `@codeCoverageIgnoreEnd`.
    - **РАЗРЕШЕНО**: применять ТОЛЬКО для кода, который невозможно протестировать (например, защитный код от багов фреймворка или конфигурация или 100% рабочая функция фреймворка).

**СТРОГО ЗАПРЕЩЕНО:**

- Обычные комментарии `// ...`, `/* ... */`.
- `TODO` в коде (используй `TODO.md` по процессу).
- Закомментированный код.
- "Украшения" и разделители кода.

**ЗАПРЕЩЕНО ДАЖЕ ЕСЛИ:**

- Нужно объяснить “почему так сделано”.
- Комментарий “очевидный и короткий”.
- Это временная пометка “позже исправить”.
- “Для новичков в команде”.

## ✅ ОБЯЗАТЕЛЬНЫЕ ТРЕБОВАНИЯ

### Static Analysis (PHPStan Level 9)

- **Zero Errors**: код должен проходить `make analyze` без ошибок.
- **Strict Mixed**: запрещено использовать `mixed` без проверок (`is_string`, `instanceof`).
- **No Unsafe Casts**: запрещено `(string)$mixed` без гарантии скалярности.
- **Generics**: используй `array<string, mixed>`, `iterable<User>` в PHPDoc.

### Переводы (i18n)

- **SSOT**: любой новый ключ (ошибки, уведомления, UI) = обязательный перевод в `messages/` (ru + en).

### Ошибки (Presentation)

- **Никакой тишины**: ошибки маппинга/валидации в Presentation должны логироваться (класс + сообщение) и быть видимыми в UI (errorSummary или поле).

### Именование

- **Классы**: `UserService`, `BookRepository`, `CreateBookCommand`
- **Методы**: `camelCase`, глагол + объект: `createBook`, `findBySlug`
- **Константы**: `UPPER_SNAKE_CASE`
- **Суффиксы**: `*Service`, `*Repository`, `*Controller`, `*Command`, `*DTO`, `*Exception`

## 🛠 ИНСТРУМЕНТЫ И СРЕДА

- **Docker-Only**: весь код выполняется в контейнерах. Запрещено запускать `php`, `composer` или `yii` локально.
- **Make-First**: используй ТОЛЬКО команды `make` (`make test`, `make dev`).
- **Makefile Usage**:
  - **Command-First**: запрещено читать `Makefile` для поиска команд. Используй только `make help`.
  - **Edit-Only**: чтение и редактирование `Makefile` разрешено ТОЛЬКО для дебага существующих или добавления новых команд.
- **Исключение**: по прямому запросу пользователя агент может писать команды `docker compose`, но не запускать их.
- **Разрешение на запуск**: если пользователь явно разрешил запуск, агент может выполнять команды `docker compose` только в рамках запроса и без побочных действий.
- **Project Map**: используй `make tree` для изучения структуры модулей и слоев.
- **LLM Navigator**: используй `llms.txt` для поиска документации и схем данных.
- **No Local State**: не полагайся на локальные файлы, не входящие в репозиторий.

## 🤖 WORKFLOW AI-АГЕНТА

### 1. Генерация и рефакторинг (цикл разработки)

1. **Читай** контракт и задачу.
2. **Планируй** изменения (KISS, YAGNI).
3. **Реализуй** код (PHP 8.5, strict_types, типизация).
4. **Цикл разработки** (повторяй сначала в конце каждого логического блока работ и после исправления ошибок в командах ниже):
   - Запусти `make dev` → исправь ошибки если есть.
   - Запусти `make test` → исправь если красное.
   - Запусти `make analyze` → исправь если красное.
5. **Самоконтроль**: перепроверь критические запреты (особенно комментарии).
6. **Запрос проверки**: попроси пользователя проверить выполнение задачи.
7. **Финальная проверка** (строго ПОСЛЕ одобрения пользователя):
   - Запусти `make test-full` → исправь если красное (это объединённый запуск: unit + integration + coverage). Также доведи coverage до 100% измененных файлов в соответствии с контрактом (не тестировать фреймворк, геттеры и т.д.).
8. **Только если всё зеленое** → готово к сдаче.

### 2. Правило MD и CSS файлов

Если меняешь **только** `.md` файлы (документацию) или `.css` файлы - запускать `make dev` и `make test` **НЕ НУЖНО**.

## 🧪 ТЕСТИРОВАНИЕ

- **Тесты пишутся вместе с кодом** - не после.
- **Один тест = один сценарий**.
- **Arrange-Act-Assert** - чёткая структура.
- **Мокай внешние зависимости**.
- **Code Coverage = 100%**:
  - Считается сумма Unit + Integration (видно в `make test-full` → `make cov`).
  - Используй `@codeCoverageIgnore` для кода, который физически невозможно достичь (защита от багов PHP/Yii) или для тестирования самого фреймворка.

## 🏛️ АРХИТЕКТУРА (Clean + Yii2)

### Слои (Deptrac Rules)

`Presentation` -> `Application` -> `Domain` <- `Infrastructure`
**Проверка**: `make deptrac` (запускать при изменении зависимостей).

### Компоненты

- **Command**: иммутабельный объект с входными данными.
- **UseCase**: оркестрация бизнес-логики (один публичный метод `execute`).
- **Repository**: скрывает ActiveRecord. Доступ только через интерфейс.
- **ActiveRecord**: только persistence (хранение). Никакой бизнес-логики.
- **Form**: валидация входных данных (из UI/API).

### Правила

- **AR = только persistence**, без бизнес-логики.
- **Repository скрывает AR** от бизнес-логики.
- **UseCase работает через Repository**, не напрямую с AR.
- **Избегай**: бизнес-логики в AR, прямого доступа к AR в контроллере, Query building в UseCase.

## 🎯 SENIOR-LEVEL PRACTICES

- **Boy Scout Rule**: оставь модуль чище, чем нашел. Увидел мелкую грязь рядом с задачей - убери.
- **SOLID, DRY, KISS, YAGNI**.
- **Dependency Inversion**: завись от интерфейсов.
- **Tell, Don't Ask**: объект сам выполняет действия.
- **Value Objects** вместо примитивов.
- **Named constructors** для ясности.
